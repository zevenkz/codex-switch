# Codex Switch Auth Backend Design

## Overview

This spec defines the first real backend-enabled version of `Codex Switch` inside the current Tauri repository. The existing UI preview remains the front-end shell, but account data will stop using mock content and instead come from real local Codex authentication data plus an app-managed multi-account store.

The core product behavior is:

- `Add account` opens an OpenAI authorization URL generated by this app
- the app listens on a local callback URL and completes the OAuth exchange itself
- newly authenticated accounts are saved into a local account store without immediately switching the active Codex account
- switching remains a separate user action that writes the chosen account back to `~/.codex/auth.json`
- quota data is refreshed from OpenAI for each saved account
- the macOS app includes a menu bar status item for quick open, refresh, switch, and quit actions

This design intentionally avoids coupling account acquisition to `codex login`. The app owns the add-account flow, while account switching stays explicit and separate.

## Goals

- Replace front-end mock accounts with real locally managed Codex accounts
- Read the current active Codex login from `~/.codex/auth.json`
- Add accounts through an app-owned OAuth browser flow with a localhost callback listener
- Persist multiple Codex accounts locally inside the app
- Keep `Add account` and `Switch account` as separate actions
- Refresh quota data from OpenAI for saved accounts
- Add a macOS status bar menu tailored to Codex account switching
- Add a simple rudder-style app icon and status bar icon

## Non-Goals

- No automatic account switching immediately after successful login
- No dependence on `codex login` for the primary add-account flow
- No automatic Codex restart after switching
- No cloud sync for saved accounts
- No Windows/Linux tray-specific UX optimization in this phase
- No broad reuse of the repository's existing provider-management model for this feature

## Key Product Rules

### Account Acquisition And Switching Are Separate

- `Add account` only adds and stores a new account
- the currently active Codex account is not changed by successful OAuth completion
- `Switch account` is the only action allowed to overwrite `~/.codex/auth.json`
- after switching, the UI will state that restarting Codex is required for the change to take effect

### Codex Live Auth Remains The Runtime Source

- `~/.codex/auth.json` remains the live file that Codex itself reads
- the app-managed account store acts as the durable multi-account library
- the app must always be able to reconstruct the live `auth.json` for the selected saved account

## Architecture

The implementation will add a focused Codex-account subsystem rather than trying to map these accounts onto the repository's generic provider architecture.

### Layers

- Rust domain and persistence for Codex accounts
- Rust OAuth callback and token-exchange flow
- Rust commands for front-end interaction
- React hooks and API bindings for accounts, switching, and refresh
- macOS tray/status-bar integration backed by the same account state

### Source Of Truth

- local Codex account store: all saved accounts and quota snapshots
- live `~/.codex/auth.json`: current active runtime account
- front-end state: derived from backend responses and emitted events

## OAuth Add-Account Flow

### Browser Authorization

When the user clicks `Add account`, the app will:

1. generate PKCE values:
   - `state`
   - `code_verifier`
   - `code_challenge`
2. start a temporary localhost HTTP listener on a fixed or managed callback port
3. build the OpenAI authorization URL in the same family as the user-provided example
4. open the authorization URL in the system browser

### Callback Handling

The local listener will wait for:

- `GET /auth/callback?code=...&state=...`

Validation rules:

- the returned `state` must match the generated state
- if the callback contains an OAuth error, the flow is failed and surfaced to the UI
- the listener must shut down cleanly after success, timeout, or cancellation

### Token Exchange

After receiving a valid authorization code, the app will:

1. exchange the code for tokens using the generated PKCE verifier
2. extract identity information from the token payload and user profile claims
3. construct a Codex-compatible auth snapshot
4. save the new account into the local account store

The app will not write the new account into the live Codex auth file at this stage.

### Account Identity Extraction

The app should derive as much identity as possible from the OAuth response and token claims:

- email
- OpenAI/ChatGPT account ID
- plan type
- user display name when available
- organizations when available

The current machine's observed `auth.json` structure shows these values are commonly available in token claims, so the first version can extract from JWT payloads without requiring an extra profile round-trip when claims are already present.

## Local Account Store

### Storage Model

Add a dedicated persisted store for Codex accounts managed by this app. This should live in the app's existing persisted storage area rather than in `~/.codex`.

Recommended record shape:

- `id`
- `email`
- `account_id`
- `plan_type`
- `display_name`
- `avatar_seed`
- `added_at`
- `last_used_at`
- `is_active`
- `auth_json`
- `quota`
- `metadata`

### Auth Snapshot

`auth_json` should store the full Codex-compatible auth object needed to reconstruct `~/.codex/auth.json`, including:

- `auth_mode`
- `last_refresh`
- `tokens.access_token`
- `tokens.refresh_token`
- `tokens.id_token`
- `tokens.account_id`
- any other required fields observed in the active file shape

This avoids lossy reconstruction when switching later.

### Bootstrapping Existing Account

On startup, the backend will:

1. read `~/.codex/auth.json` if it exists
2. parse it as the current live account
3. extract account identity
4. upsert it into the local account store if not already present
5. mark the matching stored account as active

This ensures users who already logged into Codex before installing `Codex Switch` still see their current account immediately.

## Live Account Switching

### Switch Behavior

When the user chooses a different saved account:

1. load the saved account's `auth_json`
2. atomically write it to `~/.codex/auth.json`
3. update local store flags so only that account is active
4. emit events for the front-end and tray to refresh
5. show a message that restarting Codex is required

### Safety Rules

- switching must be atomic
- the previous live auth file should be recoverable on write failure
- deleting a non-active account must not affect the current live auth file
- the currently active account cannot be deleted

## Real-Time Auth Monitoring

The app will watch `~/.codex/auth.json` for changes while running.

### Use Cases

- detect external login changes initiated outside this app
- detect external Codex account switches
- keep the app's active-account view in sync with the real live auth file

### Behavior

When the file changes:

1. debounce rapid duplicate events
2. re-read and parse the file
3. upsert the resulting account into the local account store
4. mark it active
5. emit an account-state-changed event
6. refresh tray content

This watcher is also a fallback if the OAuth callback flow succeeds but post-exchange persistence or live-file reconciliation needs confirmation.

## Quota Refresh

### Data Model

Each stored account will keep the latest quota snapshot:

- `five_hour_percent`
- `five_hour_reset_at`
- `week_percent`
- `week_reset_at`
- `refreshed_at`
- `last_error`

### Refresh Sources

The backend will use the saved account tokens to fetch the latest OpenAI quota information. The first version may reuse or adapt the repository's existing Codex subscription-reading logic where possible, but it must evolve from a single-current-account implementation into a per-saved-account implementation.

### Refresh Triggers

- manual refresh button in the main UI
- tray menu refresh action
- optional startup refresh for the active account

### Error Handling

- quota failures must not delete or invalidate the account
- the UI should preserve the last successful quota snapshot when a refresh fails
- token expiry should be surfaced distinctly from network failure

## Front-End Integration

### Main Accounts View

The existing `Codex Switch` UI will stop using mock accounts and instead render backend data.

Account card changes:

- email
- plan
- active state
- quota snapshot
- enable/switch action for non-active accounts
- delete action for non-active accounts

### Add Account UX

Clicking `Add account` will:

1. call a Tauri command to start the OAuth flow
2. open the browser
3. show a transient pending state in the UI
4. update automatically when the backend receives and stores the new account

### Settings

Language and theme behavior stays as-is from the preview implementation.

## Status Bar Menu

### Menu Structure

The macOS status item menu should be purpose-built for this feature:

- `Open Codex Switch`
- `Refresh Quota`
- separator
- one item per saved account, labeled by email
- separator
- `Quit`

### Account Menu Rules

- the active account entry is disabled and visually marked
- non-active accounts are clickable and trigger switch
- refreshing quota updates both tray labels and the main window view

### App Visibility

Clicking `Open Codex Switch` should show and focus the main window.

## macOS Iconography

### App Icon

- white background
- simple rudder graphic
- minimal, geometric, and legible at small and medium sizes

### Status Bar Icon

- monochrome template-style rudder
- optimized for macOS menu bar rendering
- should follow the repository's existing tray icon loading path for macOS template assets

## Persistence And Security

### Local Storage

Saved account data should live under the app's normal data/config directory, not in the user-visible Codex config folder.

### Sensitive Data

The implementation should treat stored tokens as sensitive. The first version may store them in the app's local persistent store if needed for product completion, but the design should keep the storage layer isolated enough to move secrets into macOS Keychain later without changing the React or tray layers.

### Logging

- never log raw access tokens, refresh tokens, or id tokens
- redact authorization URLs when logging query strings that include sensitive parameters
- redact callback query payloads where appropriate

## Failure Modes

### OAuth Problems

- browser never completes login
- callback listener port already in use
- returned state mismatch
- token exchange failure
- malformed callback payload

For all of these, the user should get a clear, non-destructive error and the pending add-account state should be cleared.

### File Problems

- `~/.codex/auth.json` missing
- parse failures for malformed auth JSON
- write failure during switch
- external edits arriving while the app is open

The app should preserve the last known saved accounts even if the live file becomes unreadable.

## Testing

### Rust Tests

- parse existing Codex auth JSON into internal account models
- build Codex-compatible auth snapshots from stored accounts
- OAuth callback state validation
- switch writes live auth atomically
- deleting inactive account preserves active account
- watcher-driven upsert of external auth changes
- tray menu generation from saved account state

### Front-End Tests

- renders backend-provided accounts instead of mock data
- add-account flow shows pending state and resolves after backend event
- switch action updates active state messaging
- delete confirmation only appears for non-active accounts
- refresh action updates quota display

### Manual Verification

- add a new account through browser login and confirm it appears without switching
- switch to a stored account and verify `~/.codex/auth.json` changes
- restart Codex and confirm the switched account is effective
- use tray menu to open app, refresh, switch, and quit
- verify menu bar icon and app icon render correctly on macOS

## Risks And Constraints

- The authorization URL pattern is based on an observed working flow and user-provided reference rather than a fully documented public desktop OAuth contract. The implementation should isolate URL generation and token exchange logic behind one module so it can be updated if OpenAI changes the flow.
- Storing multiple refreshable account snapshots increases security sensitivity; token handling must stay tightly scoped and redacted in logs.
- `auth.json` may evolve over time, so parsing should preserve unknown fields where possible instead of normalizing aggressively.
- The repository already has provider and tray infrastructure, but this feature should stay isolated enough that Codex account-switching behavior does not accidentally change generic provider features.
